## The problem this addresses

<Warning>
  Features ship faster than they get documented. The result: KAM learns about a change from a customer
  question, customers discover behaviour by trial and error, and caterers miss workflow updates.
</Warning>

The fix is not "write more docs once" — it's a **repeatable place and format** where a feature isn't
considered done until its user-facing change is written down for the audiences it affects. The
[release communication](/it/feature-changelog) page demonstrates that pattern.

## How "internal" stays internal

This pilot is published **publicly** on Mintlify's free tier, so everything here is deliberately
**non-sensitive sample content**. A real internal rollout would gate this tab behind authentication
(SSO / password), which is a paid Mintlify capability — captured as a cost line in the evaluation.

<Info>
  Source material that would feed a real version of this tab already exists: `docs/mcp/` in the app
  repo, the `egora-kam-assistant/knowledge/` markdown, and ~110 working notes in the gitignored
  `/plans/` directory.
</Info>
