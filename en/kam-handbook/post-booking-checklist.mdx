Once a request hits **DATE_CONFIRMED (17)** and an order exists, eGora tracks a
**4-point KAM checklist per order** (in the OrderHub). Each point is an email that needs to go out.

<Steps>
  <Step title="Order confirmation → customer">
    Field: `order_confirmation_sent_customer`
  </Step>
  <Step title="Order confirmation → caterer">
    Field: `order_confirmation_sent_caterer`
  </Step>
  <Step title="Delivery note → caterer">
    Field: `delivery_note_sent`
  </Step>
  <Step title="Invoice → customer">
    Field: `invoice_sent`
  </Step>
</Steps>

## How it behaves (system facts)

<CardGroup cols={2}>
  <Card title="Created automatically" icon="wand-magic-sparkles">
    Created automatically when the request transitions to **DATE_CONFIRMED** (with a fallback that creates
    it as soon as the offer→order link is established).
  </Card>
  <Card title="Needs a linked order" icon="link">
    The order must be linked to a catering request.
  </Card>
  <Card title="Syncs to HubSpot" icon="rotate">
    Changes to the checklist are automatically synced to HubSpot.
  </Card>
  <Card title="Look it up, don't assume" icon="magnifying-glass">
    Whether a point is done can only be determined by looking the order up via the connector.
  </Card>
</CardGroup>

<Warning>
  The assistant can only remind you which of the four points are still open **if it looks the order up** —
  it must never assume a point is done.
</Warning>

<Note>
  This 4-point checklist and the [status labels](/en/kam-handbook/request-lifecycle) are the pieces that are
  **enforced by the system**. Everything beyond that in this handbook is a working description of how we
  work, and is meant to be refined with the team's real SOPs.
</Note>
