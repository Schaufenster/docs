---
title: "When a request goes sideways"
description: "The right terminal statuses to keep reporting clean when a request doesn't convert."
icon: "triangle-exclamation"
---

When a request isn't going to convert, set the **correct terminal status** so reporting and
conversion metrics stay clean. Use the real status, not a catch-all category.

| Situation | Status to set |
| --- | --- |
| Caterer can't deliver it | **REJECTED_BY_CATERER (4)** |
| Customer says no | **REJECTED_BY_CUSTOMER (7)** |
| Customer cancels later | **CANCELLED_BY_CUSTOMER (8)** |
| Customer goes silent | **LOST_NO_RESPONSE (9)** |
| Lost for any other reason | **LOST_REQUEST (16)** |
| eGora declines | **DENIED_BY_EGORA (10)** |

<Note>
  These belong to the *rejected / lost* status group (4, 7, 8, 9, 10, 16) in the
  [lifecycle](/en/kam-handbook/request-lifecycle).
</Note>

<Warning>
  This is about **request status hygiene**, not a formal escalation SOP. eGora currently has no single,
  system-defined "escalation process" — escalations live in scattered working documents. If you need
  the current state, ask the assistant to check **Notion** (it will tell you honestly if no consolidated
  draft exists, rather than invent one).
</Warning>
