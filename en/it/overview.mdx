## The problem we're tackling

<Warning>
  Features ship faster than they get documented. The result: KAM learns about a change from a customer question, customers discover behavior by trial and error, and caterers miss workflow updates.
</Warning>

The fix isn't "write more docs once" — it's a **repeatable place and format** where a feature is only considered done once its user-facing change is written down for the affected audiences. The [Release communication](/it/feature-changelog) page demonstrates this pattern.

## How "internal" stays internal

This pilot is published **publicly** on Mintlify's free tier, so everything here is deliberately **non-sensitive example content**. A real internal rollout would put this tab behind authentication (SSO / password) — that's a paid Mintlify feature and is captured as a cost line item in the evaluation.
