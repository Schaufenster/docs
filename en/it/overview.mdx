## The problem this is solving

<Warning>
  Features ship faster than they're documented. The result: KAM hears about a change through a
  customer question, customers discover behavior by trial and error, and caterers miss workflow
  updates.
</Warning>

The fix isn't "write more docs, one more time" — it's a **repeatable place and format** in which a
feature is only considered done once its user-facing change is written down for the affected
audiences. The [Release communication](/en/it/feature-changelog) page demonstrates this pattern.

## How "internal" stays internal

This pilot is published **publicly** on the free Mintlify plan, so all content here is deliberately
**non-sensitive sample content**. A real internal rollout would hide this tab behind authentication
(SSO / password) — a paid Mintlify feature, captured as a cost item in the evaluation.
