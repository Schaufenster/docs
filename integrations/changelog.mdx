Die ursprüngliche Lücke, auf die diese KB abzielt: **Features werden ausgeliefert, ohne kommuniziert zu werden** — an KAM, Kunden und
Caterer. Die Changelog-Automatisierung schließt sie.

## Funktionsweise (aktuelles Setup — Route 1)

Während des Tests nutzen wir die **nativen Automatisierungen** von Mintlify:

<CardGroup cols={2}>
  <Card title="Aktualisierung aus Code-Änderungen" icon="code">
    Erzeugt Entwürfe für Inhalts-Updates, wenn sich Produkt-/Feature-/API-Quellcode ändert.
  </Card>
  <Card title="Changelog-Entwurf" icon="list">
    Erzeugt regelmäßig Changelog-Entwürfe aus den jüngsten Produkt-Updates.
  </Card>
</CardGroup>

Jeder Entwurf landet als **Pull Request** in `Schaufenster/docs` und aktualisiert
[Release-Kommunikation](/it/feature-changelog). Ein Mensch prüft und merged → der Eintrag erreicht sowohl die KAMs
(Web) als auch Claude (MCP).

## Hilf nach: der `Changelog:`-Trailer

Wenn ein Commit für Nutzer relevant ist, füge eine einzeilige Trailer-Zeile hinzu, damit die Automatisierung ein sauberes Signal hat:

```
feature for the salutation style

Changelog: Customers can now be addressed Du/Sie; KAM sets it per customer.
```

Das ist optional, schärft aber den Entwurf — und sorgt für eine saubere Historie, falls wir die Engine irgendwann wechseln.

## Das gewünschte Format

Jeder Eintrag sagt, **was sich geändert hat**, **wer betroffen ist** (KAM / Kunden / Caterer) und **was zu tun ist** —
siehe die Beispiele unter [Release-Kommunikation](/it/feature-changelog).

<Note>
  Route 1 läuft auf den kostenpflichtigen Agent-Credits von Mintlify (während des Tests im Trial). Falls das zu einer Einschränkung wird,
  wechseln wir auf eine credit-freie Custom-Pipeline (eine GitHub Action + ein Claude-`changelog`-Skill), die **dieselbe**
  Changelog-Datei schreibt — sodass nachgelagert nichts angepasst werden muss.
</Note>
